# GAP-Step 完整实验操作手册

本文档说明如何在当前项目中运行一套相对完整的 GAP-Step 实验。默认使用已有 conda 环境 `isaac`，不再创建新的 `gap-step` 环境。

## 1. 实验前检查

进入项目目录：

```bash
cd /home/jack/wyh/时变窗口
conda activate isaac
```

检查 PyTorch 和 CUDA：

```bash
python -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'CPU')"
```

检查基础测试：

```bash
pytest -q
```

如果测试通过，应看到：

```text
5 passed
```

## 2. 清理旧结果

如果要从零开始实验，先删除运行产物：

```bash
rm -rf data checkpoints logs runs
```

这些目录只保存数据、checkpoint、日志和可视化结果，不包含源码。

## 3. 快速烟测流程

先跑一遍短流程，确认环境没有问题：

```bash
python scripts/render_random_policy.py
python trainers/generate_demos.py
python trainers/train_bc.py
python trainers/train_ppo.py
python trainers/train_ppo.py --init none --output checkpoints/visual_ppo.pt
python trainers/evaluate.py
python scripts/render_trained_policy.py --checkpoint checkpoints/bc_aux.pt
```

注意：默认配置是快速可运行版本，不代表完整收敛实验。完整实验请按后续章节提高数据量和 PPO 更新次数。

## 4. 推荐完整实验顺序

建议按难度从 E1 到 E3 逐步跑：

| 场景 | 配置 | 目的 |
|---|---|---|
| E1 | `configs/env_e1.yaml` | 单窗口固定开放，验证控制和碰撞逻辑 |
| E2 | `configs/env_e2.yaml` | 单窗口周期变化，验证穿越时机 |
| E3 | `configs/env_e3.yaml` | 双窗口异步变化，验证窗口选择 |

不要一开始只看 E3。E3 失败时，需要先确认 E1/E2 是否稳定成功。

## 5. 为每个场景建立训练配置

当前训练脚本通过 yaml 控制数据路径、环境配置、checkpoint 目录和日志目录。建议为 E1/E2/E3 分别创建配置文件，避免结果互相覆盖。

```yaml
train_split: 0.9
```

## 7. 训练 BC-only 和 BC+Aux

分别训练三个场景：

```bash

python trainers/train_bc.py --config configs/train_bc_e2.yaml
```

建议先评估 BC 结果。如果 BC 在 E1 都不成功，先不要跑 PPO，应先检查环境、教师演示和 BC loss。

## 8. PPO 微调配置

PPO 是最容易不稳定的部分。默认 `20` updates 只适合烟测。完整实验建议至少几百到上千 updates。

### 8.1 E1 PPO 配置

创建 `configs/train_ppo_e1.yaml`：

```yaml
env_config: configs/env_e1.yaml
seed: 10
device: auto
init_checkpoint: checkpoints/e1/bc_aux.pt
output_checkpoint: checkpoints/e1/bc_aux_ppo.pt
total_updates: 300
rollout_steps: 1024
minibatch_size: 256
epochs: 4
lr: 0.00003
gamma: 0.99
gae_lambda: 0.95
clip_ratio: 0.1
value_coef: 0.5
entropy_coef: 0.001
aux_coef: 0.1
max_grad_norm: 0.5
runs_dir: runs/e1
```

### 8.2 E2 PPO 配置

## 9. 训练 PPO

先从 BC+Aux 初始化微调：

```bash
python trainers/train_ppo.py --config configs/train_ppo_e1.yaml
python trainers/train_ppo.py --config configs/train_ppo_e2.yaml
python trainers/train_ppo.py --config configs/train_ppo_e3_full.yaml
```

再训练 Visual-PPO baseline，也就是不从 BC 初始化：

```bash
python trainers/train_ppo.py --config configs/train_ppo_e1.yaml --init none --output checkpoints/e1/visual_ppo.pt
python trainers/train_ppo.py --config configs/train_ppo_e2.yaml --init none --output checkpoints/e2/visual_ppo.pt
python trainers/train_ppo.py --config configs/train_ppo_e3_full.yaml --init none --output checkpoints/e3/visual_ppo.pt
```

注意：Visual-PPO 从零开始通常更难训练，短训练失败是正常的。它主要作为对照组。

## 12. TensorBoard 查看训练曲线

启动 TensorBoard：

```bash
tensorboard --logdir runs --port 6006
```

浏览器打开：

```text
http://localhost:6006
```

重点看：

- BC train/val loss 是否下降；
- PPO return_mean 是否上升；
- PPO reward_mean 是否严重下降；
- BC+Aux 的 width/safe loss 是否明显优于 BC-only。

## 13. 如何判断实验是否成功

优先看 E1：

- `Teacher-Heuristic` success_rate 应接近 1.0；
- `BC-only` 或 `BC+Aux` 应接近 1.0；
- 如果 E1 失败，说明基础控制或训练数据有问题。

再看 E2：

- `BC+Aux` 的 `width_mae` 应明显低于 `BC-only`；
- `safe_f1` 应明显更高；
- success_rate 应不低于 BC-only，理想情况下更稳定。

最后看 E3：

- `gate_choice_accuracy` 应较高；
- `closed_gate_attempt_rate` 应低；
- `BC+Aux` 应比 `BC-only` 有更好的 width/safe 预测；
- `BC+Aux+PPO` 理想上应提高 return 或 success_rate。

如果 PPO 后结果变差，说明 PPO 破坏了 BC 策略。此时不要认为方法失败，应先降低 PPO 学习率和探索强度，或保存 best checkpoint。

## 14. 当前实现的已知限制

当前 PPO 是最小可运行版本，不是完整稳定 RL 框架：

- 只优化连续加速度；
- gate head 不参与环境动作；
- 没有 best checkpoint 选择；
- 没有 KL early stopping；
- 没有 action std annealing；
- 没有 vectorized environments；
- 没有 curriculum 自动切换；
- 没有评估回滚机制。

因此完整实验中，BC/BC+Aux 是更可靠的第一阶段结果；PPO 需要更长训练和更保守超参数。

## 15. PPO 不稳定时的处理

如果 `BC+Aux+PPO` 比 `BC+Aux` 差，按顺序尝试：

1. 把 `lr` 从 `0.00003` 降到 `0.00001`；
2. 把 `clip_ratio` 从 `0.1` 降到 `0.05`；
3. 把 `entropy_coef` 设为 `0.0`；
4. 把 `aux_coef` 提高到 `0.2`；
5. 先只在 E1/E2 上训练 PPO，确认不退化后再训 E3；
6. 暂时使用 `bc_aux.pt` 作为主结果，把 PPO 作为后续改进项。
